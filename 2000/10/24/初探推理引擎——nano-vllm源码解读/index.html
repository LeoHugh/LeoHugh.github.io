初探推理引擎——nano-vllm源码解读

发表于2000-10-24|cs
|浏览量:

概览

在开始阅读源码前，我们需要知道评价一个推理引擎的指标有哪些：

  • Time To First Token (TTFT)：首字延迟（决定用户的等待耐心）。

  • Time Per Output Token (TPOT)：每个输出词延迟（决定生成速度的流畅度）。

  • Throughput (Tokens/second)：单位时间处理的 Token 吞吐量（最重要的商业指标，决定机器成本）

nanovllm 的代码架构极其精简，完美复刻了 vLLM 的核心思想。整体目录如下：

整个推理引擎可以拆解为两大核心命题：

  1. server system: token 调度与显存管理 (engine/)
  2. excution engine: 具体模型的物理计算 (layers/)

接下来，我们分别讲述这两个命题

server system：token调度与显存管理

continous batching

传统的深度学习的server system，是request level级别的调度。它会把一定数量的request打包进一个batch，发送
给excution engine。等excution全部算完后，再返回给server system，开启下一轮batch。
这样的调度存在三大问题:

  • early-finished 的request需要等待最慢的request finish才能吐出，缺乏一个将已生成结束的请求从Batch中移除并提前返回结果的机制
  • late-join的request必须要等待当前一轮的batch结束，缺乏一个实时将request插入batch的机制
  • 空间浪费.对于较短的seq需要做padding来补齐.

为了解决这三大问题，Orca: A Distributed Serving System for Transformer-Based Generative Models 提出了continous batching.其将server system的调度算法从request level精细化到token level.
其想法是，对于batch每生成一批next tokens，就做一次check，丢弃完成的request，加入新的request.
可以看下一下的动图辅助理解

在采用continous batching后:

  • early-finished的延迟就降到了最慢的一个request的next token生成时间
  • late-join也可以在每轮迭代后有机会进入batch
    另外一个问题是，在硬件底层上是如何实现continous batching的呢?
    在excution中，传统的intputs=[batch_size,max_seq_length,hidden_size]
    而在continous batching中，inputs=[tokens_nums,hidden_size],并通过一个position数组来维护每个seq的位置。
    总结:可以认为采用了类似cpu流水线化的思想，在底层实现上类似于hpc中压缩稀疏行的存储格式
    局限之处:我们可以看见原论文的思想并没有区分prefill与decode阶段，也没有针对kvcache优化.

阶段感知调度算法

在nano-vllm中，实现的是相对粗糙的prefill优先调度算法。
我们一共有两个队列:waiting 队列和 running队列,调度以step为最小单位
在每轮，我们会优先检查waiting队列是否仍然存有seq，只要有，就会放进batch。我们设置了三个约束条件:

  • num_seqs < self.max_num_seqs：并发请求数不能超载。
  • num_batched_tokens + len(seq) > self.max_num_batched_tokens：单次处理的 Token 总数不能炸显存（Prefill 阶段 Token 量极大，必须严格限制）。
  • self.block_manager.can_allocate(seq)：底层的 KV Cache 管理器必须有足够的空闲物理块（Block）来容纳这个新请求
    当不做prefill时，才会做decode。
    在生成每一个token后，我们会向底层的 KV Cache 申请一个新的位置。如果当前 Block 满了，就需要申请新的物理 Block
    如果显存不够，先尝试踢掉队尾的，或者踢掉自己。

在实际中，我们会使用chunk prefill或者pd分离等技术

BLOCK_manager：page attention && prefix cache

面对巨大的kvcache显存占用，使用连续内存分配会导致大量的内部碎片和外部碎片。
为了解决显存碎片化的问题，我们引入了page attention。

首先，我们定义了一个基础数据结构block

我们维护了一个free_block_list,used_block_ids,seq.blobk_table,hash_to_block的对应

这里的block_manager是在逻辑层面解决分配问题,实际的物理分配在model_runner中
prefill阶段的分配，目标是给每个block的tokens找一个对应的blockId.
同时，借鉴了区块链的做法，通过哈希实现了一个严格的prefix cache 复用

may_allocate是在decode阶段按需求增加block
比方说一个block满了，那要做哈希并放到hash_to_block
比方说新来一个token，那要新开一个block给他

回收block，这里就是做了一个简单的惰性回收，采用了lru算法.这个问题也是开放的。

总结

从一个朴素的推理循环出发，逐步引入了：

  • 两阶段执行：区分 Prefill（计算密集）和 Decode（访存密集）
  • 动态调度：Continuous Batching，抢占机制保证前进性
  • 页式 KV Cache：按需分配，
  • 链式哈希实现 Prefix Cache
  • CUDA Graph：降低 Decode 阶段的 Kernel Launch 开销
  • 张量并行：多卡推理，列并行 + 行并行
  • 组合算子优化：torch.compile 融合，Triton 自定义 Kernel
    这里面还有部分我没搞懂，后续仍会修改补充~
