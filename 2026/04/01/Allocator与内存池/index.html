一、STL Allocator 源码剖析

1. Allocator 的本质: Thin Wrapper

首先，我们先深入 GCC 标准库 libstdc++ 的源码，一探 allocator 类的真实面目 。

可以看见，allocator 类继承自 allocator_base 类（即 new_allocator 类），它仅仅是一层薄封装 (Thin Wrapper) 。同时，它是无状态的：

  • using is_always_equal = true_type;
  • friend _GLIBCXX20_CONSTEXPR bool operator==(const allocator&, const allocator&) _GLIBCXX_NOTHROW { return true; }

3. 核心枢纽: allocator_traits

事实上，容器 (如 vector) 并不会直接与 allocator 交互，而是通过一个交接层 alloc_traits.h 。它的源码主要做了三件事：

  • 类型推导: 如果没有指定 pointer 类型，则自动使用 alloc 默认类型 。
    using pointer = _detected_or_t<value_type*, __pointer, _Alloc>;
  • 行为兜底 (SFINAE): 检查 Allocator 是否有自定义的 construct，如果没有，则使用默认的 placement new 。
    • 通过定义检测器 _construct_helper 检查 Alloc 是否有名为 construct 的函数 。
    • 在实际调用的入口 _S_construct 中，如果 Allocator 自己写了 construct 则调用它；否则使用默认的 placement new 或 C++20 的 std::construct_at 。
  • 重绑定 (Rebind):
    template <typename T> using rebind_alloc = typename Alloc::template rebind<T>::other;
  • 特化优化: 针对 std::allocator 做了特化处理 。
  • 状态传播: 默认情况下都是 false，但在 STL alloc 下允许 move 时直接”偷”指针 。
    • using propagate_on_container_move_assignment = true_type;

二、现代 Allocator 的思考

  1. 为什么没有内存池?
    在目前版本的 std::allocator 实现中没有内存池，是因为内存管理任务已经下沉到 malloc 了，allocator 已完全沦为一个分配的接口层 。
  2. Allocator 层的作用是什么?
    • 空间配置器: 可以轻松切换到栈分配或共享内存分配策略 。
    • 类型安全: 给裸内存空间加上了对象的类型 。
    • 生命周期的管理: 明确分开了内存分配与对象构造 。

三、重铸经典: 手写 SGI 风格 Allocator

笔者尝试实现的其实是一个老版本的 SGI allocator 。

1. 为什么要自己写 Allocator?

从结果来看确实快了。笔者认为提升的原因在于：

  • 更好的缓存局部性
  • 减少多线程下的开销 (如果是单线程池)
  • 减少元数据开销: deallocate 接口要求传大小，省去了 malloc 存储 cookie 的头部开销 。
  • 降低逻辑复杂度

2. 架构设计

  • 前端 (MyAllocator): 符合 C++ 标准库规范的接口层，将请求转发给后端 。
  • 后端 (Memory Pool): 实际的内存管理者，采用单例模式 。

四、核心机制详解

1. 自由链表数组 (Free List Array)

采用了类似哈希表的桶机制：

  • 机制: 维护一个指针数组 freeList[32] 。
  • 对齐: align = 8 。
  • 映射: 0号链表管理 8 字节块，1号管理 16 字节块…直到 31 号管理 256 字节块 。
  • 计算下标公式: (n+7)/8 - 1 。

2. 嵌入式指针 (Embedded Pointer)

利用空闲内存块的前 8 个字节存储 next 指针，节省空间 。

这也决定了 64 位系统下最小区块必须是 8 字节 。

3. 内存池的水源管理 (chunk_alloc)

内存从大池子里切分的逻辑分三种情况：

  • 情况 A: 内存池水量充足，直接切分 20 个块 。
  • 情况 B: 水量不足但能提供至少 1 个块，计算最大能提供的块数并全切出来 。
  • 情况 C: 内存池彻底干枯。
    1. 处理残渣: 将残余内存 (如 4 字节) 挂到对应的 freeList 上 。
    2. 注入活水: 调用 ::operator new 向 OS 申请新内存，使用 SGI 扩容公式：2 * demandSize + (heapSize >> 4) 。
    3. 递归: 申请到内存后，重新尝试分配 。

五、待完成与展望

  • 线程安全: 目前是单例且无锁的，多线程环境下需要加锁 。
  • 内存归还: 目前只增不减，没有实现将内存还给 OS 的机制 (典型的空间换时间) 。

六、正确性验证与性能分析

测试显示 MyAllocator 在复杂嵌套容器 (vector<vector<int>>) 及自定义对象中表现正常 。

性能测试结果 (std::allocator vs MyAllocator):

测试项目 STL (us) Pool (us) Ratio (STL/Pool)
List<int> Push Back (500k) 21304 4632 4.60x
List<Struct64> Push Back (500k) 26190 18068 1.45x
Map<int, int> Insert (200k) 32169 22029 1.46x
List Random Push/Pop (500k) 16682 5605 2.98x
Vector<int> Push Back (1M) 2896 1672 1.73x

数据解读：

  • List: 提升最明显 (4.6x)。无元数据开销且节点物理连续，带来了极高的 Cache Hit 率 。
  • 大对象: 随着对象变大，构造/拷贝时间占比增加，分配优势递减。说明内存池适合小而多的对象 。
  • 随机操作: freeList 的 $O(1)$ 指针操作在处理频繁释放时极其高效 。
  • Vector: 即使是连续内存，在元素较少、内存块较小的初始阶段也能享受到红利 。

总结

在单线程、小对象、高频分配的场景下，SGI 风格的内存池具有统治级的性能优势 (最高 4.6 倍提升) 。虽然现代 malloc 很强，但在“特化”的路上，手写 Allocator 依然有效 ！
