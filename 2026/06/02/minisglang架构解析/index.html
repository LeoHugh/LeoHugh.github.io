mini-sglang源码解析

基本介绍

mini-sglang的两大特点是
轻量级，仅5千多行代码
高性能，支持核心性能优化技术:

  • cuda graph
  • Overlap Scheduling
  • chunk prefill
  • radix cache

用户模块

用户与系统交互的入口点。minisglang提供了两个http端点

  • /generate
  • /v1/chat/completions

client模块

负责 HTTP 协议处理、全局状态管理、进程编排

api_server.py

提供openai风格的api接口
使用fastapi解析http请求，
使用ptdantic进行数据类型强约束
请求路由与转换

异步任务管理
采用了一个FrontendManager的进程全局单例:全局计数，请求状态管理，清理过期请求；发送消息到tokenizer，收回tokenizer的消息，按uid发消息；流式响应；清理资源
使用了zmq的通道模式进行通信
流式响应处理
与后端通信

  • launch.py：进程启动器，负责 spawn 各个 worker 进程

    args.py

    配置管理模块，解析命令行参数，完成后端服务配置
    一些参数
    基本的参数
    — model-path — model-source huggingface/modelscope
    — dtype int8/f16/f32
    —dummy-weight 测试模式

并行参数
—tensor-parallel-size 张量并行度
—disable-pynccl # 禁用PyNCCL通信库

服务网络配置
— host
— port

kvcache管理
— memory-ratio 分配的显存比例
— num-pages 最大页面数
— page-size 页面大小
— cache-type 缓存策略

请求调度与并发
—max-running-requests # 最大并发请求数
—max-prefill-length # chunk size
—cuda-graph-max-bs # CUDA Graph最大批次大小

进程拓扑(微服务模式)
—num-tokenizer # Tokenizer进程数（0=共享模式）

注意力机制优化
—attention-backend # Attention实现后端（可多选）
—moe-backend # MoE层实现后端

schedule进程

核心调度

  • 资源分配（kv cache，token indices）
  • 请求调度与批次管理
  • 协调各个管理器（Prefill/Decode/Cache/Table）

engine模块
实际执行模型推理：

  • Engine：推理引擎，协调各组件
  • Model：模型前向传播
  • Attention：注意力计算（可插拔后端）
  • KV Cache：键值对存储
